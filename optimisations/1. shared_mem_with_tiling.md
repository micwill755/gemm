# CUDA Matrix Multiplication Tiling Concept

## Overview

Tiling (also called blocking) is a fundamental optimization technique that breaks large matrix operations into smaller, manageable chunks that fit efficiently in GPU shared memory.

## The Problem: Memory Hierarchy

### GPU Memory Hierarchy (Speed & Size)
```
Registers     →  Fastest,  ~64KB per SM,     Per-thread
Shared Memory →  Fast,     ~96KB per SM,     Per-block  
L1 Cache      →  Medium,   ~128KB per SM,    Per-SM
L2 Cache      →  Slow,     ~6MB total,      Global
Global Memory →  Slowest,  ~24GB total,     Global
```

### Naive Approach Problem
```cuda
// Each thread computes one dot product
for (int k = 0; k < K; k++) {
    sum += A[row * K + k] * B[k * N + col];  // 2K global memory accesses
}
```

**Issues:**
- Every access goes to slow global memory (400+ cycles)
- Same data read multiple times by different threads
- No data reuse between threads

## The Solution: Tiling Strategy

### Core Concept
**Break the computation into small tiles that fit in shared memory, enabling data reuse.**

### Visual Representation

For matrices A(64×64) × B(64×64) = C(64×64) with TILE_SIZE=16:

```
Matrix A (M×K = 64×64):
     K dimension
   ┌─────┬─────┬─────┬─────┐
M  │ t=0 │ t=1 │ t=2 │ t=3 │  ← Split K into tiles
   │16×16│16×16│16×16│16×16│
   └─────┴─────┴─────┴─────┘

Matrix B (K×N = 64×64):
K  ┌─────┐
   │ t=0 │ 16×16
   ├─────┤
   │ t=1 │ 16×16  
   ├─────┤
   │ t=2 │ 16×16
   ├─────┤
   │ t=3 │ 16×16
N  └─────┘

Result C (M×N = 64×64):
   ┌─────┬─────┬─────┬─────┐
   │16×16│16×16│16×16│16×16│
   ├─────┼─────┼─────┼─────┤
   │16×16│16×16│16×16│16×16│
   ├─────┼─────┼─────┼─────┤
   │16×16│16×16│16×16│16×16│
   ├─────┼─────┼─────┼─────┤
   │16×16│16×16│16×16│16×16│
   └─────┴─────┴─────┴─────┘
```

## Algorithm Breakdown

### Step 1: Thread Block Organization
```cuda
dim3 blockDim(TILE_SIZE, TILE_SIZE);  // 16×16 = 256 threads per block
dim3 gridDim((N + TILE_SIZE - 1) / TILE_SIZE,
             (M + TILE_SIZE - 1) / TILE_SIZE);
```

Each thread block computes one 16×16 tile of the result matrix C.

### Step 2: Shared Memory Allocation
```cuda
__shared__ float As[TILE_SIZE][TILE_SIZE];  // 16×16 tile from A
__shared__ float Bs[TILE_SIZE][TILE_SIZE];  // 16×16 tile from B
```

**Memory usage**: 2 × 16² × 4 bytes = 2KB per block (well within 96KB limit)

### Step 3: Tile Loop Structure
```cuda
int numTiles = (K + TILE_SIZE - 1) / TILE_SIZE;  // Ceiling division
for (int t = 0; t < numTiles; t++) {
    // Load tiles
    // Compute partial results
    // Accumulate
}
```

### Step 4: Collaborative Tile Loading

#### Loading A Tile
```cuda
int aCol = t * TILE_SIZE + threadIdx.x;
As[threadIdx.y][threadIdx.x] = A[row * K + aCol];
```

**What each tile iteration loads:**
- t=0: Columns 0-15 of A
- t=1: Columns 16-31 of A  
- t=2: Columns 32-47 of A
- t=3: Columns 48-63 of A

#### Loading B Tile
```cuda
int bRow = t * TILE_SIZE + threadIdx.y;
Bs[threadIdx.y][threadIdx.x] = B[bRow * N + col];
```

**What each tile iteration loads:**
- t=0: Rows 0-15 of B
- t=1: Rows 16-31 of B
- t=2: Rows 32-47 of B  
- t=3: Rows 48-63 of B

### Step 5: Computation Using Shared Memory
```cuda
__syncthreads();  // Ensure all threads have loaded their data

for (int k = 0; k < TILE_SIZE; k++) {
    sum += As[threadIdx.y][k] * Bs[k][threadIdx.x];  // Fast shared memory
}

__syncthreads();  // Prepare for next tile
```

## Memory Access Analysis

### Naive vs Tiled Comparison

| Aspect | Naive | Tiled | Improvement |
|--------|-------|-------|-------------|
| Global Memory Reads | 2K per thread | 2K/16 per thread | **16x reduction** |
| Memory Reuse | None | 16x per element | **16x improvement** |
| Memory Latency | 400+ cycles | ~20 cycles (shared) | **20x faster** |
| Cache Efficiency | Poor | Excellent | **Coalesced access** |

### Data Reuse Visualization

**Naive**: Each thread reads independently
```
Thread (0,0): A[0,0] A[0,1] A[0,2] ... A[0,63]  ← 64 reads
Thread (0,1): A[0,0] A[0,1] A[0,2] ... A[0,63]  ← Same 64 reads!
Thread (0,2): A[0,0] A[0,1] A[0,2] ... A[0,63]  ← Same 64 reads!
```

**Tiled**: Collaborative loading + reuse
```
All 16 threads in row 0: Load A[0,0-15] once → reuse 16 times
All 16 threads in row 0: Load A[0,16-31] once → reuse 16 times
All 16 threads in row 0: Load A[0,32-47] once → reuse 16 times
All 16 threads in row 0: Load A[0,48-63] once → reuse 16 times
```

## Performance Benefits

### 1. Reduced Global Memory Traffic
- **Before**: Each element read K times across all threads
- **After**: Each element read once per tile, reused within block

### 2. Improved Memory Coalescing
- **A matrix**: Perfect coalescing (consecutive addresses within warp)
- **B matrix**: Partially coalesced (some stride patterns remain)

### 3. Better Cache Utilization
- Shared memory acts as a software-managed cache
- Data locality improved through blocking

### 4. Scalable Performance
- Tile size can be tuned for different GPU architectures
- Larger tiles = more reuse (up to shared memory limits)

## Implementation Considerations

### Tile Size Selection
```cuda
#define TILE_SIZE 16  // Common choices: 8, 16, 32
```

**Trade-offs:**
- **Larger tiles**: More data reuse, but higher shared memory usage
- **Smaller tiles**: Less memory pressure, but more tile iterations
- **Optimal size**: Depends on GPU architecture and matrix dimensions

### Boundary Handling
```cuda
if (row < M && aCol < K)
    As[threadIdx.y][threadIdx.x] = A[row * K + aCol];
else
    As[threadIdx.y][threadIdx.x] = 0.0f;  // Padding for irregular sizes
```

### Synchronization Points
```cuda
__syncthreads();  // Critical for correctness
```
- Required after loading tiles
- Required before loading next tiles
- Ensures all threads see consistent shared memory state

## Real-World Performance Impact

### Benchmark Results (DeepSeek-V3 Scale)
| Matrix Size | Naive (ms) | Tiled (ms) | Speedup |
|-------------|------------|------------|---------|
| 2048×4096   | 162.3      | 78.9       | **2.06x** |
| 4096×5120   | 529.6      | 264.4      | **2.00x** |
| 8192×6144   | 1756.8     | 885.1      | **1.98x** |

### Why 2x (not 16x) Speedup?
1. **Memory coalescing**: Not perfect for B matrix
2. **Compute bound**: Eventually limited by arithmetic throughput
3. **Overhead**: Synchronization and tile management costs
4. **Cache effects**: L1/L2 caches provide some benefit even in naive version

## Next Optimization Opportunities

1. **Tensor Cores**: Use specialized matrix hardware (4-10x additional speedup)
2. **Double Buffering**: Overlap computation with memory transfers
3. **Register Tiling**: Further reduce shared memory accesses
4. **Mixed Precision**: Use FP16 for higher throughput

---

*Tiling transforms memory-bound operations into compute-bound operations by maximizing data reuse through the GPU memory hierarchy.*
